# HTTP File Upload Design

## Goal

Extend the existing HTTP/API debugger with standard `multipart/form-data` requests so a request can contain text fields and one or more local files without changing the behavior of the existing JSON, URL-encoded form, text, or empty body modes.

## Scope

The feature adds a `multipart` body mode with ordered rows. Each row is enabled or disabled, has a field name, and is either a text value or a local file. Multiple file rows are supported, including repeated field names.

The feature does not add folder uploads, drag and drop, request collections, upload progress, file previews, or automatic conversion of the existing URL-encoded form mode.

## User Interface

The Body type selector gains a `multipart/form-data` option. In that mode, the request editor displays an ordered row editor consistent with the existing key-value editors.

Each multipart row contains:

- An enabled toggle.
- A field name.
- A row type selector with Text and File options.
- A text input for text rows.
- A read-only selected-file display and a system file-picker action for file rows.
- The existing row removal behavior.

The editor always keeps one empty row available. Switching a row between Text and File clears the value from the previous type so a hidden local path or stale text value cannot be sent accidentally.

The Send action is blocked when an enabled multipart row has no field name or when an enabled File row has no selected file. File-selection cancellation leaves the current row unchanged.

## Request Contract

The frontend contract gains a multipart body mode and a dedicated row type:

```ts
type HttpBodyMode = "none" | "json" | "form" | "multipart" | "text";
type HttpMultipartFieldKind = "text" | "file";

interface HttpMultipartField {
  id: string;
  enabled: boolean;
  key: string;
  kind: HttpMultipartFieldKind;
  value: string;
  filePath: string;
  fileName: string;
}
```

`HttpDebugRequest` gains `multipartFields: HttpMultipartField[]`. The field order is preserved from the editor through the Rust request builder. `value` is used only for Text rows; `filePath` and `fileName` are used only for File rows.

The initial request contains one empty Text multipart row. Existing requests and history records that do not contain `multipartFields` are normalized to an empty/default multipart row when loaded so previously stored data remains usable.

## File Selection and Native Boundary

The frontend uses Tauri's native file dialog and stores the selected path only in the current in-memory request. File bytes are not loaded into JavaScript and are not serialized as base64. Rust reads the selected file when sending and streams it through `reqwest` multipart support.

This keeps large uploads outside the WebView memory path. A missing, unreadable, or changed file produces a request error while keeping the editor state intact.

The multipart `Content-Type` header and boundary are generated by `reqwest`. If the user has an enabled manual `Content-Type` header while multipart mode is selected, validation blocks sending and explains that ToolDock must manage this header. This avoids missing or mismatched boundaries.

## Rust Request Construction

The Rust request model mirrors the multipart row contract. After the existing URL, timeout, and header validation, multipart validation checks enabled rows before starting the request.

For each active row in order:

- Text rows add a regular multipart text part.
- File rows open the local path and create a streaming file part.
- The visible filename comes from the selected file name and is sent with the part.

The existing HTTP method, query parameter, header, redirect, TLS, timeout, response-size, and response-presentation behavior remains unchanged.

## History and Privacy

History keeps multipart row order, enabled state, field name, row type, and non-sensitive text values. Sensitive text fields use the existing name-based redaction rules.

Before persistence, every File row has `filePath` and `fileName` cleared. Restoring history therefore recreates the row structure but requires the user to select each file again. File paths and file contents are never written to HTTP history.

An enabled restored File row without a selected path blocks sending and displays the same missing-file validation message as a new row.

## cURL Generation

The current request can still be copied as cURL:

- Text rows use `--form` with `field=value`.
- File rows use `--form` with `field=@local-path`.
- Disabled or incomplete rows are omitted.
- Windows output keeps the existing PowerShell 7-safe quoting rules.
- macOS output keeps the existing POSIX-shell-safe quoting rules.

cURL generation uses only the current in-memory request. Restored history does not generate file arguments until files have been selected again.

## Error Handling

- Missing multipart field name: block Send and identify the row.
- Missing selected file: block Send and identify the row.
- File missing or unreadable at send time: return a concise file-read error and preserve the request.
- Manual `Content-Type` in multipart mode: block Send and ask the user to remove it.
- File dialog cancellation: make no state change.
- HTTP or network failure after upload starts: use the existing request error handling and preserve the request.

## Localization

Chinese and English labels are added for the multipart body mode, Text/File row types, file selection, missing field name, missing file, unreadable file, and managed `Content-Type` message.

## Verification Coverage

When implementation verification is requested, focused frontend coverage should check multipart draft validation, history normalization, privacy projection, and PowerShell/macOS cURL generation. Focused Rust coverage should check text/file multipart construction, repeated field names, generated boundaries, missing files, and history redaction. Existing body modes must remain unchanged.

## Acceptance Criteria

- A multipart request can combine text and local file rows.
- Multiple files and repeated field names are sent in editor order.
- File bytes are streamed by Rust rather than loaded into frontend memory.
- ToolDock generates the multipart boundary and prevents conflicting manual `Content-Type` headers.
- Current requests generate correct multipart cURL arguments.
- HTTP history never stores local file paths or file contents.
- Restored file rows require explicit file reselection before sending.
- Existing JSON, URL-encoded form, text, and empty body modes continue to behave as before.
